Реализация тестов с использованием Java-RestAssured. (рекомендуется версия Java - 17 и выше; среда разработки IntelliJ IDEA). Написать пробные запросы к сервису DaData ( https://dadata.ru/api/ ).
Вот задачи, которые были поставлены:
1. Зарегистрироваться для получения Token (бесплатно доступно 10000 запросов в день)
2. Выбрать GET & POST эндпоинты для проверки. Например GET https://dadata.ru/api/v2/detectAddressByIp?ip=  и POST https://suggestions.dadata.ru/suggestions/api/4_1/rs/suggest/address
3. Реализовать тесты этих эндпоинтов.
4. Поместить код тестов в репозиторий.

Ответ:
Основные проблемы и рекомендации
1. Ошибка в статус-коде для невалидного токена
В тесте testDADATAUsageInvalidToken ожидается 401, однако DaData при неверном токене возвращает 403.
Исправление: заменить responseSpecification401() на responseSpecification403() (как уже сделано в testIPLocateInvalidToken).
2. Потенциальные NullPointerException
В testIPLocate перед вызовом userData.getPostal_code().isEmpty() и аналогичных методов нет проверки на null. Если поле отсутствует в ответе – упадёт тест.
Исправление: заменить assertFalse(...isEmpty()) на assertNotNull(...) и затем assertNotEquals("", ...).
3. Отсутствие тестов для POST-запросов
По заданию требовалось реализовать и GET, и POST эндпоинты. В коде присутствуют только GET-запросы (даже для /suggest/address используется GET).
Рекомендация: добавить тесты для POST-запроса /suggest/address (например, с телом { "query": "Москва" }).
4. Неполная проверка статистики
В testDADATAUsage проверяются только поля верхнего уровня (date, services, remaining). Поля внутри Services и Remaining остаются без проверки.
Улучшение: добавить проверки, что они не null, и, возможно, что значения >= 0.
5. Неоднозначная обработка EnvLoader
Метод get() удаляет точку с запятой в конце строки – это нестандартное поведение и может быть избыточным.
Рекомендация: убрать эту манипуляцию или оставить только trim().
6. Дублирование кода
В каждом тесте повторяются блоки given().spec(...).header(...).... Это можно вынести в отдельные методы-фабрики для конкретных эндпоинтов.
Предложение: создать класс RequestFactory, который будет возвращать готовый RequestSpecification с уже установленным токеном.
7. Слабая проверка в тестах на пустой/короткий запрос
Тесты testSuggestAddressEmptyQuery и testSuggestAddressShortQuery проверяют только статус 200, но не анализируют содержимое.
Рекомендация: извлечь ответ и проверить, что suggestions – пустой массив.
8. Разрозненность тестов
Все тесты собраны в одном классе APITest. При росте количества проверок это усложнит навигацию.
Предложение: разбить на отдельные классы: IpLocateTest, SuggestAddressTest, StatsTest.
9. Несоответствие версии Java
В README.md указана Java 15, а в pom.xml – Java 17.
Исправление: синхронизировать документацию.
10. Отсутствие проверки формата даты
В DADATAStats поле date проверяется только на null. Можно добавить проверку на соответствие формату (например, yyyy-MM-dd).
